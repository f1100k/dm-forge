# 0002. PostgreSQL + Prisma with versioned migrations

**Status:** accepted
**Date:** 2026-05-02
**Deciders:** Founding team
**Supersedes:** —

## Context

dm-forge needs a relational database for users, sessions, AI connections (BYOK), campaigns, and (later) the full Codex/Arcs/Scenes graph. SRD search will use Postgres FTS (`docs/architecture-overview.md` → SRD search). Tiptap content lives as `Jsonb`. We want a typed ORM that doesn't fight the schema and a migration trail that works in CI from day one.

## Decision

- **PostgreSQL 16+** as the only supported database. No SQLite fallback for dev — Docker Compose ships a one-line Postgres at `localhost:5432` so dev parity matches prod.
- **Prisma** as the ORM. Schema lives in `packages/db/prisma/schema.prisma`. Generated client emitted to `packages/db/src/generated/client` and re-exported via `@dm-forge/db`.
- **`zod-prisma-types`** generator runs alongside the Prisma client to produce Zod schemas from the model definitions. This avoids hand-writing duplicate validation for tRPC inputs.
- **Migrations are versioned** (`prisma migrate dev` in dev, `prisma migrate deploy` in CI/prod). The `migrations/` folder is committed. No "db push" against shared environments.
- **Client IDs only.** Schema fields are `id String @id` — never `@default(cuid())`. IDs are generated by `@dm-forge/db`'s `createId()` (cuid2), called on the client before any mutation. See `docs/coding-patterns.md`.
- **Soft delete** for campaign entities (`state EntityState @default(ACTIVE)`). Hard delete only for non-campaign records (sessions, ephemeral logs).
- **`postinstall` hook** in `packages/db` runs `prisma generate` so the typed client is available immediately after `pnpm install`.

## Consequences

- **Positive:** one schema source of truth; migrations are reviewable diffs; FTS, `Jsonb`, partial indexes are all available when needed; types flow from DB → tRPC → frontend with zero hand-typing.
- **Negative:** Prisma's generator step is part of every install (small extra time); developers must remember to `prisma migrate dev` when pulling schema changes.
- **Neutral:** Postgres-only locks us out of SQLite-based test setups. Tests that need a DB will use the Docker Postgres or testcontainers (decided per feature).

## Alternatives considered

- **Drizzle** — closer to raw SQL, no codegen, but the Zod integration and migration ergonomics are weaker than Prisma's at this stage.
- **Server-generated UUIDs** — incompatible with the optimistic-mutation pattern (Constitution principle 4); the client needs the ID before the request returns.
- **SQLite for dev** — rejected; FTS, `Jsonb`, GIN indexes diverge from prod and would mask bugs.
